# DevPortal Parity Design: algokit-utils-ts

## Goal

Bring algokit-utils-ts docs to devportal-ready parity with algokit-utils-py. The output is a `devportal-docs.tar.gz` release asset containing markdown content, sidebar config, and manifest metadata.

## Key Architectural Difference

Unlike utils-py (Sphinx-based API docs generated by `api-build.sh`), utils-ts uses `starlight-typedoc` — an Astro integration that generates API markdown during `astro build`. API docs don't exist on disk until the build runs.

## Architecture

```
Library repo (algokit-utils-ts)            Devportal repo
├── docs/                                  ├── .github/actions/
│   ├── package.json (standard scripts)    │   └── publish-docs/
│   ├── normalize-links.typedoc.ts         │       └── action.yml (composite)
│   ├── build-sidebar-json.ts              │
│   └── sidebar.config.ts                  └── (triggers rebuild on dispatch)
│
├── .github/workflows/
│   └── publish-devportal-docs.yml         ← thin caller of composite action
```

### Principle: Standardized Scripts

Every library repo exposes the same `package.json` script names. The devportal composite action calls these generically — it doesn't need to know whether the repo uses Sphinx or TypeDoc.

**Standard scripts in `docs/package.json`:**

| Script | Purpose |
|--------|---------|
| `build` | Astro build (triggers starlight-typedoc for API docs) |
| `normalize-links` | Prepend site base to relative markdown links |
| `build-sidebar` | Generate `dist-devportal/sidebar.json` |
| `build-devportal` | Orchestrates: build → normalize-links → build-sidebar |

This means adding a new generator type (e.g., rustdoc) only requires implementing these scripts in the new repo — no changes to the composite action.

## Changes to algokit-utils-ts

### 1. Add dependencies (`docs/package.json`)

- `starlight-links-validator` — catches broken links at build time
- `tsx` (dev) — runs TypeScript scripts (normalize-links, build-sidebar)

### 2. Wire up starlight-links-validator (`docs/astro.config.mjs`)

```ts
import starlightLinksValidator from 'starlight-links-validator'

// In starlight() plugins array:
starlightLinksValidator({
  errorOnInvalidHashes: false,
  errorOnLocalLinks: false,
})
```

### 3. Create normalize-links.typedoc.ts

Adapted from `normalize-links.sphinx.ts`. Processes markdown files to convert relative links to absolute paths rooted at the site base (`/algokit-utils-ts/`).

**Targets:**
- `guides/` — hand-written content, with filesystem fallback for broken relative links
- `api/` — starlight-typedoc generated content, no fallback (links are Starlight slugs)

**Core functions (same as sphinx variant):**
- `resolveRelativePath` — collapse `.` and `..` components
- `slugExists` — check if a slug maps to a real content file
- `buildFileIndex` — index all content files for fallback lookups
- `findBestMatch` — score candidates by trailing path component overlap
- `normalizeLinksInContent` — regex-based link replacement

**CLI:** reads `base` from `astro.config.mjs`, supports `--base` flag override.

### 4. Add standardized scripts (`docs/package.json`)

```json
{
  "normalize-links": "npx tsx normalize-links.typedoc.ts",
  "build-sidebar": "npx tsx build-sidebar-json.ts",
  "build-devportal": "npm run build && npm run normalize-links && npm run build-sidebar"
}
```

### 5. Housekeeping

- Delete `build-sidebar-json.mjs` (superseded by `.ts` version)
- Add `dist-devportal/` to `docs/.gitignore`
- Fix typo in `index.mdx`: `quick-starta` → `quick-start`

### 6. Thin calling workflow (`.github/workflows/publish-devportal-docs.yml`)

```yaml
on:
  push:
    tags: ['v*']
  workflow_dispatch:

jobs:
  publish-docs:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '22' }
      - uses: pnpm/action-setup@v4
      - run: pnpm install --frozen-lockfile
      - run: pnpm --prefix docs install
      - uses: algorandfoundation/devportal/.github/actions/publish-docs@main
        with:
          devportal-dispatch-token: ${{ secrets.DEVPORTAL_DISPATCH_TOKEN }}
```

## Composite Action Design (devportal repo)

**Location:** `algorandfoundation/devportal/.github/actions/publish-docs/action.yml`

**Inputs:**

| Input | Required | Default | Description |
|-------|----------|---------|-------------|
| `docs-dir` | no | `docs` | Path to docs directory |
| `devportal-dispatch-token` | yes | — | PAT for triggering devportal rebuild |

**Steps performed by the action:**

1. Run `pnpm --prefix <docs-dir> run build-devportal`
2. Generate `dist-devportal/manifest.json` (repo, ref, tag, timestamp)
3. Create `devportal-docs.tar.gz` from `dist-devportal/`
4. Publish to GitHub Release:
   - Tag push → attach to version release
   - Otherwise → upsert `docs-latest` pre-release
5. Trigger `docs-published` repository_dispatch on devportal repo

**Note:** Checkout, Node setup, pnpm setup, and dependency installation live in the calling workflow because composite actions cannot use `uses:` to reference external actions.

## Future: npm Package

Once the composite action is stable, the library-side scripts (normalize-links, build-sidebar) can be published as an npm package. Library repos then update their `package.json` scripts to call the published package instead of local scripts.
